### 技术选型

在移动端有`ionic`、`react-native`和`weex`三大移动端跨平台大法，也相对应`angular`、`react`和`vue`。那么在桌面端也有`electron`和`nw`两大桌面端跨平台大法。

**Electron**和**Nw**是使用`HTML`、`CSS`和`JS`构建跨平台桌面应用且集成`Node`授予页面访问底层系统权限的技术方案。

使用`electron`开发的成功案例有`VSCode`、`Atom`、`Github`、`Slack`、`WordPress`和`微信`等等。使用`nw`开发的成功案例有`钉钉`和`微信开发工具`，没几个，笔者也想不出来了。

`electron`相对于`nw`有以下优点和缺点，而`nw`快凉了，就不再探索。

- 优点
	- 核心扩展较易：基于`Node技术`开发，由于`node-gyp`逐渐人性化导致`C++`和`JS`混用较易
	- 界面定制较强：基于`Web技术`开发，GUI核心使用`Chromium`，JS引擎使用`V8`
	- 开发成本较低：熟练使用`HTML+CSS+JS`的前端开发者都能开发，结合各种UI框架省事又省心
	- 技术方案更稳：有大团队的技术支持，运行更稳定，缺陷更少，只要浏览器调试没问题基本就没问题了
	- 迭代更新更快：方便热更新，下载新代码覆盖旧代码
- 缺点
	- 运行速度较弱：应用运行起来可能会卡也可能会慢，使用了`Webkit内核`的锅，参考`Chrome`即可
	- 进程管理较傻：除了启动主进程控制整个应用，还需启动一些辅助进程完成某些工作
	- 构建包体较大：相当于打包了一整个`Chromium`，不大才怪

不多说，凡是要开发桌面应用通通选`electron`吧。

另外，Web框架选择`react`。本来笔者想选择`vue`体验最新的`vue 3`特性，但是`vue 3`正式版迟迟未出，就放弃了。还是使用平常非常喜欢的`react hooks`开发吧，爱不释手。

不想搞那些平平无奇的抽奖特效，想搞一些酷炫屌炸天的抽奖特效，就选择`three`这个WebGL框架。想制造酷炫吊炸天特效，除了`canvas`，`webgl`也能助你一臂之力。

所以本教程的最终组合是`Electron + React Hooks + Three`，妥妥的。

**另外提醒一下，最好有接触过React，无经验者可到React官网稍微了解再来学习本教程会更佳。本教程是完完全全的实战教程，不会详细讲解Electron/React/Three的深入细节和某个API怎样用，若有不懂可自行百度或在底部评论等笔者解答。**

### 依赖安装

接下来需安装本教程的主角`electron`，在安装过程中若不规范上网可能100%安装失败，所以安装前需填个坑。

众所周知`NPM`的服务器在国外，而某些模块的部分依赖又托管在国外某些服务器上，即使执行`npm config set registry https://registry.npm.taobao.org/`将源镜像切换到淘宝镜像，在安装过程中遇上部分依赖，其下载地址还是指向托管在国外某些服务器上的地址，导致安装依赖失败。所以需找出这些依赖的下载地址，将其切换到淘宝镜像上。

在安装`electron`前需执行以下命令方可安装。若对解决这些镜像问题感兴趣，可回顾笔者发表的[《NPM镜像那些险象环生的坑》](https://mp.weixin.qq.com/s/9DJKOGYiv7JqqlltPipVqg)，让你彻底摆脱所有`NPM安装依赖`的坑。

```sh
# 切换淘宝镜像
npm config set registry https://registry.npm.taobao.org/
npm config set disturl https://npm.taobao.org/mirrors/node/
npm config set electron_mirror https://npm.taobao.org/mirrors/electron/
# 安装 electron
yarn add electron
```

本教程使用的`electron`为最新版本：`10.0.0+`。

### 进程管理

`electron`可让我们使用`JS`调用丰富的`原生操作系统API`创造桌面应用。可把它看作是一个`Node环境`的变体，只是它专注于桌面应用，而`Node`专注于服务应用。

这不意味着`electron`是一个单调的`GUI`库。相反，`electron`使用`Web`创建页面展示图形用户界面，使用`Node`授予页面访问底层系统权限，所以可把它看作是一个被`JS`控制的精简版`Chrome`。

上述分析中可知`electron`的精华，**使用Web创建页面展示图形用户界面，使用Node授予页面访问底层系统权限**。因此派生出两个主要概念，**主进程**和**渲染进程**。

> 主进程

**主进程**指`electron`运行`package.json`的`main.js`而产生的进程。在主进程中运行的脚本通过创建页面展示图形用户界面。**一个应用总是有且只有一个主进程**。

> 进程区别

主进程使用`BrowserWindow实例`创建页面。每个`BrowserWindow实例`都在自己的渲染进程里运行页面。当一个`BrowserWindow实例`被销毁后，对应的渲染进程也会被终止。

主进程管理所有页面以及其对应的渲染进程。每个渲染进程独立且唯一，只关心它所运行的页面。

在页面中调用与GUI相关的原生API是不被允许的，因为在页面里操作原生GUI资源会非常危险，且容易造成资源泄露。若想在页面里使用`GUI操作`，其对应的渲染进程必须与主进程通讯，请求主进程进行相关的`GUI操作`。

> 渲染进程

**渲染进程**指`electron`运行`Web`创建页面而产生的进程。由于`electron`使用了`Chromium`展示页面，所以`Chromium`的多进程架构也被使用到。**一个应用可有多个渲染进程**。

### 入口构建

在每次构建`Electron应用`前，得先执行`bruce b`并选择对应的执行环境构建`React应用`。简单来说就是先让`React应用`存在，再让`Electron应用`引用`React应用`。

有了上述概念性的理解，首先在根目录下创建一个`main.js`。

> 区分执行环境和配置构建脚本

常规项目都会将执行环境分为`开发环境`、`测试环境`和`生产环境`，所以使用`npm scripts`构建项目时需配合`cross-env`声明`process.env.NODE_ENV`，在`main.js`中通过`process.env.NODE_ENV`区分执行环境而执行不同任务。

以下统一声明执行环境关键字`process.env.NODE_ENV`，`bruce-cli`也使用了该关键字。

- **开发环境**：`dev`
- **测试环境**：`test`
- **生产环境**：`prod`

在`开发环境`中以调试的方式开发应用，使用`bruce-cli`就像往常那样构建`React应用`，使用`electron`启用应用的主线程和渲染线程。在`测试环境`和`生产环境`中以打包的方式输出应用，使用`electron-packager`根据需求打包应用。

`electron-packager`相关命令参数如下，依据需求可组合出打包命令：`electron-packager . Lottery --all --overwrite --out ./dist/app --icon ./favicon.ico --electron-version 10.0.0`。

- **sourcedir**：应用根目录，即`package.json`所在目录，使用`.`表示
- **name**：应用名称，使用`<name>`
- **platform**：构建平台，可选`darwin`、`linux`、`mas`、`win32`，全选使用`--all`表示
- **overwrite**：是否覆盖输出，是则使用`--overwrite`表示
- **distdir**：打包输出路径，使用`--out <dir>`表示
- **icon**：应用图标，使用`--icon <dir>`表示
- **version**：构建版本，需和项目当前`electron`版本一致，使用`--electron-version <ver>`表示

接下来将上述理论知识代码化。

```sh
# 安装 cross-env 和 electron-packager
yarn add cross-env electron-packager -D
```

在`package.json`中插入以下配置。

```json
{
	"scripts": {
		"build:prod": "cross-env NODE_ENV=prod electron-packager . Lottery --all --overwrite --out ./dist/app/prod --icon ./favicon.ico --electron-version 10.0.0",
		"build:test": "cross-env NODE_ENV=test electron-packager . Lottery --all --overwrite --out ./dist/app/test --icon ./favicon.ico --electron-version 10.0.0",
		"dev": "cross-env NODE_ENV=dev electron ."
	}
}
```

接下来编写整个应用的入口文件`main.js`。在开发阶段，页面以浏览器内存的形式存在，简单来说就是一个可访问的地址。在测试阶段和生产阶段，页面以实体文件的形式存在，简单来说就是`webpack`打包输出的文件。所以`BrowserWindow实例`需在不同情况使用不同的方法引用页面资源。

- **loadURL()**：引用页面地址
- **loadFile()**：引用页面文件

`bruce-cli`存在三种构建模式。

- **开发环境**：页面的在线调试地址为`http://127.0.0.1:9999`，端口在启动构建进程时可自行选择
- **测试环境**：页面的打包输出路径为`dist/test`，HTML入口为`dist/test/index.html`
- **生产环境**：页面的打包输出路径为`dist/prod`，HTML入口为`dist/prod/index.html`

在开发环境下可为网页的渲染进程开启一个`Chrome DevTools`，方便后续的开发调试。在测试环境和生产环境下通过`path模块`获取对应的HTML入口。

```js
const Path = require("path");
const { app, BrowserWindow } = require("electron");

function CreateWindow() {
    const mainWindow = new BrowserWindow({
        height: 1200,
        minHeight: 1200,
        minWidth: 1600,
        width: 1600
    });
    if (process.env.NODE_ENV === "dev") {
        mainWindow.loadURL("http://127.0.0.1:9999");
        mainWindow.webContents.openDevTools();
    } else {
        const entry = Path.join(__dirname, `./dist/${process.env.NODE_ENV}/index.html`);
        mainWindow.loadFile(entry);
    }
}
```

`CreateWindow()`创建完成后，当应用初始化后监听主进程的常规事件。

```js
// Electron初始完成并创建窗口时触发
app.on("ready", CreateWindow);
// Electron内部窗口闲置点击后重新创建窗口时触发
app.on("activate", () => !BrowserWindow.getAllWindows().length && CreateWindow());
// Electron内部所有窗口关闭时触发：MacOS中除非用户操作cmd+q显式退出，否则应用与菜单栏始终处于活动状态
app.on("window-all-closed", () => process.platform !== "darwin" && app.quit());
```

`BrowserWindow`配置文档请查看[这里](https://www.electronjs.org/docs/api/browser-window#new-browserwindowoptions)。
`main.js`源码演示请查看[这里](https://github.com/JowayYoung/lottery/blob/master/main.js)。

### 调试打包

上述提到，在每次构建`Electron应用`前，得先执行`bruce b`并选择对应的执行环境构建`React应用`。简单来说就是先让`React应用`存在，再让`Electron应用`引用`React应用`。

以下分别在三种执行环境中体验一次桌面应用的构建。

> 开发环境

- 执行`bruce b`，选择`开发环境`
- 执行`npm run dev`

> 测试环境

- 执行`bruce b`，选择`测试环境`
- 执行`npm run build:test`

> 生产环境

- 执行`bruce b`，选择`生产环境`
- 执行`npm run build:prod`

就这样，三种执行环境都能完美运行，项目搭建的流程正式收官。
